# 기획서: admin-approval

## Meta
- feature: admin-approval
- type: feature
- package: apps/admin, apps/api

## 배경 / 왜
현재는 ALLOWED_DOMAINS/ALLOWED_EMAILS 환경변수로 로그인 허용 여부를 통제하고 있어, 신규 관리자를 추가하려면 운영자가 배포 과정을 거쳐야 한다. 실제 운영에서는 관리자 충원·교체가 잦고, 이를 앱 내부에서 신청·승인으로 처리해야 감사 추적과 권한 위임이 가능해진다. 또한 최초 부팅 시 누군가는 super_admin이어야 시스템이 돌아가므로 부트스트랩 규칙도 함께 정리한다.

## 목표
- 운영자가 재배포 없이 앱 내부에서 관리자를 충원·교체할 수 있다 (권한 관리의 소스 오브 트루스를 DB로 이전).
- 누구든 Google 계정으로 로그인 시도는 가능하되, 승인된 사용자만 콘솔에 진입할 수 있다.
- 미승인(pending) 사용자에게는 "승인 대기 중"임이 화면에서 명확히 전달된 뒤 세션이 종료된다.
- super_admin은 별도 화면에서 신청 목록을 보고 승인/거절할 수 있다.
- 최초 사용자는 수동 개입 없이 자동으로 active + super_admin이 된다.

## 사용자 흐름
1. 신규 사용자가 Google로 로그인을 시도하면 DB에 pending 상태로 등록된다.
2. pending 사용자는 로그인 성공 직후 "승인 대기 중"임을 알리는 토스트가 노출되고, 콘솔에 진입하지 못한 채 즉시 로그아웃된다.
3. super_admin은 관리자 목록 화면에서 전체 관리자를 상태(pending/active/rejected)별로 조회한다.
4. super_admin은 관리자 상세 화면에서 개별 신청 건을 열어 승인 또는 거절한다.
5. 승인된 사용자는 다음 로그인부터 정상적으로 콘솔에 진입한다.
6. pending 상태로 로그아웃된 사용자가 이후 승인되었더라도 기존 세션은 이미 종료되어 있으므로, 다시 로그인해야 콘솔에 진입할 수 있다.
7. 거절(rejected)된 사용자가 다시 로그인을 시도하면 상태 안내 토스트("접근이 거절되었습니다") 후 즉시 로그아웃되며, 상태는 rejected 그대로 유지된다.
8. DB에 관리자가 하나도 없는 상태에서 첫 로그인이 발생하면 해당 사용자는 자동으로 active + super_admin으로 등록된다. 동시 로그인 경합 시 먼저 커밋된 1명만 super_admin, 나머지는 pending으로 처리된다.

## 범위
### In Scope
- 관리자 레코드의 상태 모델 (pending / active / rejected) 및 역할 모델 (super_admin 포함).
- Google OIDC 로그인 시 신규 사용자를 pending으로 자동 등록.
- 미승인·거절 사용자의 콘솔 진입 차단 및 상태별 안내 토스트 후 자동 로그아웃.
- super_admin 전용 관리자 목록 화면 (상태별 조회).
- super_admin 전용 관리자 상세 화면 (승인/거절 처리).
- 최초 사용자 자동 부트스트랩 규칙: DB 관리자 0명일 때 첫 로그인 사용자를 active + super_admin으로 자동 등록.
- 기존 ALLOWED_DOMAINS/ALLOWED_EMAILS 환경변수 기반 허용 로직 제거.

### Out of Scope
- 관리자 목록/상세 화면의 시각 디자인 (Stitch 산출물 이후 별도 단계에서 부착).
- super_admin 외 세분화된 역할/권한 체계 (RBAC 확장은 별도 기능).
- 승인·거절 시 이메일/알림 발송.
- 감사 로그 UI.
- 관리자 자가 탈퇴, 역할 위임·회수, 일괄 처리.

## 미결 사항
1. 거절(rejected) 상태 되돌리기: 운영 유연성 우선 시 super_admin이 상세 화면에서 pending/active로 되돌릴 수 있도록 허용. 감사 추적 우선 시 영구 차단 유지. spec 단계에서 결정.
