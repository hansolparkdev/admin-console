# google-oidc-login — 재기획용 요약 메모

> 구 `plan.md`는 삭제됨 (critic 3회차 자동 FAIL + admin-shell 보다 후속 재기획 필요). 이 메모는 새 `/planning` 진입 시 컨텍스트로만 쓴다. plan 산출물 아님.

## 목적

`apps/admin`의 보호 라우트에 "조직 Google 계정 로그인"을 강제. 현재 BFF 프록시는 신원 없이 upstream 호출 중이라 감사·RBAC 기반이 없다.

## 핵심 기능 8개

- **F1** `/login` "Google로 로그인" → 동의 화면 → 콜백 → 보호 라우트 진입
- **F2** 로그인 성공 시 httpOnly 쿠키 세션 발급, 새로고침/탭 이동에도 유지
- **F3** Google access token 만료 시 refresh token으로 서버측 자동 갱신 (Auth.js JWT callback)
- **F4** 미로그인 상태로 보호 라우트 접근 시 `/login`으로 리다이렉트 + 로그인 후 원래 URL 복귀 (`callbackUrl`)
- **F5** 이미 로그인된 상태로 `/login` 재접근 시 보호 홈으로 리다이렉트
- **F6** BFF 프록시(`/api/*`)가 세션 access token → `Authorization: Bearer`로 변환해 `apps/api`에 전달
- **F7** 헤더 아바타 메뉴 "로그아웃" → shadcn `AlertDialog` 확인 → 세션 파기 → `/login` (취소 시 세션 유지)
- **F8** 헤더 아바타 + 드롭다운에 이름·이메일 표시 (이니셜 fallback)

## 성공 지표

- 로그인 성공률 ≥ 99%
- 로그인 리다이렉트 p95 ≤ 3초
- refresh 자동 갱신 성공률 ≥ 98%
- 보호 라우트 미로그인 접근 차단율 100%

## 확정된 기술 전제 (현 레포 인프라와 정합)

- **Auth.js v5 + Google OIDC** (core 기본). Keycloak은 on-prem 백업용.
- **세션은 httpOnly 쿠키만**. localStorage/sessionStorage 금지 (CLAUDE.md 금지 패턴).
- **통합 지점**: admin-shell의 `(app)/layout.tsx` 상단 `TODO(google-oidc-login)` hook point에 `auth()` 가드 한 줄 추가.
- **라우트 배치**: `/login`은 `(public)/login/page.tsx` (Shell 미적용). `(app)` / `(public)` 라우트 그룹 규약 준수 (`docs/rules/folder-conventions.md`).
- **BFF**: `apps/admin/src/app/api/[...proxy]/route.ts`에서 `auth()` 세션 → `Authorization: Bearer`로 변환 (현 Step 8 구현 확장).

## 범위 밖 (명시적으로 제외)

- 도메인 제한 (예: `@company.com`만 허용) → **RBAC 슬라이스**에서 도입
- RBAC 자체 (Role/Permission) → 별도 슬라이스
- 2FA / IdP 체인 / Keycloak → 별도

## 재기획 시 결정 필요

- 세션 쿠키 만료 전략 (absolute vs rolling, 기간)
- refresh 실패 시 UX (강제 로그아웃 시점, 토스트·모달 문구)
- 로그인 실패 에러 매핑 위치 (`lib/auth-error-messages.ts` 등)
- 로그인 화면 디자인 참조 소스 (`DESIGN_REF`) 와 충실도 (`DESIGN_FIDELITY`) 결정

## 착수 명령 예시

```

```

`/planning` Step 2.5에서 묻는 항목:

- `DESIGN_REF` — 로그인 화면 참조가 있으면 (Stitch/Figma/이미지/로컬), 없으면 `none`
- `DESIGN_FIDELITY` — `strict` / `guide` / `loose` / `none`
